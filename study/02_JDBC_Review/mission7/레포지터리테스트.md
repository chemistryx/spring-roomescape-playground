# Repository Layer 테스트 🏃‍♂️
이전에도 한 번 JPA 학습을 하면서 테스트에 대해서 학습해본 경험이 있다. 간단한 crud 를 실습
해보면서 테스트를 진행했다. 도메인이 적었지만, 각 도메인을 저장하는 레포지터리 레이어를 
테스트하기 위해 매우 비슷하고 반복적인 코드에 한 번 찾아보았는데, 기본적인 curd를 위해서 테스트
코드를 작성하는 것보다 커스텀 한 메서드를 위해서 테스트하는게 부담감이 더 적을 것 같다고 생각한
적이 있다.

<br/>

그래서 이번 기회에 레포지터리 레이어를 테스트하는 이유와 어떻게 해야 바람직한지 조사를 해보고 
테스트 코드를 작성해보려고 한다.

<br/>

## 데이터베이스 레이어 테스트하는 이유
> 스프링 데이터베이스 레이어 테스트는 **_애플리케이션의 코드와 데이터베이스 간 연결을 확인_**하고, 오류를 
> 찾아내기 위해 수행합니다.

<br/>

**테스트의 목적**
- 애플리케이션의 코드와 데이터베이스 간의 연결을 확인합니다.
- 오류를 찾아냅니다.
- 테스트 대상을 독립적으로 테스트합니다.
- 각 계층의 책임을 분리합니다.
- **테스트 목적에 맞는 적합한 방법을 선택**합니다.

<br/>

테스트의 목적 중에 `테스트 목적에 맞는 적합한 방법을 선택`한다. 라는 부분을 앞으로 신경써서 
테스트하는게 좋겠다.

<br/>

제일 먼저 커넥션을 잘 맺는지 테스트를 하는 방법에 대해서 알아보자. 기존의 제공된 테스트 코드에서 
커넥션을 맺는지 확인하는 테스트가 존재하는데, 참고해서 관련 자료 찾아보자.

<br/>

### 테스트 데이터베이스 연동
테스트 케이스는 src/test 에서 실행되기 때문에 src/test 에 있는 application.properties 에 기존 데이터
베이스 설정을 해야한다.

<br/>

설정 이후 테스트 케이스를 실행하면 기존의 데이터베이스 데이터로 인해서 테스트가 실패할 수 있다. 
각 테스트는 독립적이여야 하는데 테스트 디비와 운영디비가 같다면 테스트가 영향을 받을 수 있기 때문에
분리하여 운영해야 한다.

<br/>

테스트용 데이터베이스를 통해 테스트를 했을 경우에도 데이터가 저장되며 남기 때문에 
저장된 데이터들은 나머지 테스트 케이스에 영향을 준다. 이를 해결하기 위한 방법으로 
`트랜잭션`과 `롤백전략`을 사용할 수 있다.

<br/>

지금은 운영 디비도 H2 데이터베이스를 사용하고 있지만, 다른 디비를 사용하게 되었을 때
사용할 수 있을 것 같다. 조금 더 고민해보니 테스트 코드에서 사용하는 디비가 다를텐데 
운영 디비 커넥션을 어떻게 테스트해야할지 잘 모르겠다. 

일단, 데이터베이스 커넥션을 테스트하는 방법의 종류이다.

<br/>

### 데이터베이스 테스트 하기 위한 방법 

|테스트 방법|사용대상|주요 목적|
|-----------|--------|---------|
|DataSource|모든 DB환경|기본적인 커넥션 확인|
|HikariCp|HikariCp 사용 시|커넥션 풀 설정 확인|
|EntityManager|JPA 사용시|JPA 환경에서 커넥션 확인|
|JdbcTemplate|SQL 직접 실행|DB 정상 응답 확인|
|ApplicationContext|전체 빈 확인|데이터베이스 빈이 정상적으로 등록되었는지 체크|
|H2 Database|테스트 환경|인 메모리 DB로 가볍게 테스트|

<br/>

다음과 같은 방법 중에 H2 디비를 사용하고 있고, JdbcTemplate을 사용하고 있다. 그럼 어떤걸 선택할까.
일단 `@SpringBootTest`는 전체 애플리케이션 컨텍스트를 로드한다. 이에 레포지터리 레이어의 테스트
하는데는 적합하지 않다. 그 다음으로는 `@JdbcTest` 이 어노테이션은 `JdbcTemplate`과 `DataSource` 만 로드한다.

<br/>

레포지터리 빈만 로드하기 위해서 `@JdbcTest` 어노테이션을 이용하면 될 것 같다. 그리고 운영환경의 데이터베이스 
커넥션 테스트를 하기 위해서는 별다른 어노테이션을 사용하지않고, `DriverManager` 객체의 `getConnection` 메서드를
상요하여 직접 커넥션을 생성해서 테스트할 수 있다.

<br/>

`DataBaseConnectionTest` 에서 jdbc를 이용해서 커넥션을 검증하고, 운영 디비는 `DriverManager`를 이용해서 커넥션 테스트를
해보았다. 그런데 운영환경 디비는 시작을 안했는데 어떻게 성공한지 의문이다... 아니 진짜 왜 성공하지..?

<br/>

### 커넥션 테스트를 해보려고 어노테이션을 사용중 @DataJdbcTest 와 @JdbcTest 두 개의 어노테이션이 존재하는것을 확인했다.
참조한 블로그에서 글을 읽어보았는데, 잘 이해가 안간다. 일단 나중에라도 이해갈 수 있으니 주소 남겨두자.

## JdbcTest 어노테이션을 이용해서 예약 레포지터리 레이어를 테스트해보자.
앞서 글을 읽을 때 알게된 사실인데 @JdbcTest 어노테이션 안에는 @Transactional 어노테이션이 존재한다. 
즉 각각의 테스트가 하나의 트랜잭션이고, 테스트가 끝나면 rollback 이 된다는 사실을 알았다. 이에 테스트해보자.

<br/>

- `@JdbcTest` 는 Repository 빈을 자동으로 로드하지 않는다. 처음에 빈을 찾을 수 없다고 나왔다.
  - -> JdbcTemplate 관련 빈만 로드하여 @Repository 빈은 가지고 오지 않는다.
  - 해결 방법 : 
    - @Import(ReservationDatabaseRepository.class) 를 추가한다.
    - 생성자로 직접 JdbcTemplate을 주입시켜준다. 

<br/>

## 내가 레포지터리 레이어에서 id 값은 테스트 하지 않은 이유
우선 테스트가 하나씩 테스트 할 경우에는 문제가 되지 않으나, 전체로 테스트를 했을 경우에는 데이터가 롤백
되더라도 자동으로 id 값이 증가된다. id 값을 테스트 한다면 전체 테스트는 실패하게 된다. 그럼 id 값을 테스트를 해야
하는지 안해도 되는지 생각해본다. 내가 테스트하는 목적은 데이터베이스에 저장이 잘되는지 테스트 하는것이다. id 값이
올라가던 내려가던 상관이 없다고 생각을 했다. 그건 디비에서 작동하는 것이기 때문이다. 아니라면, 저장한 객체의 아이디 값과
저장 후 나온 객체의 아이디 값을 비교해서 테스트하는 방법을 사용하여 id 값의 생성도 잘 되었는지 확인할 수 있을 것 같다.
